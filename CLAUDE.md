# CLAUDE.md — Codebase Guide for AI Assistants

## What is this project?

A mobile-first web application ("Hautgesund Kochen") that generates weekly anti-inflammatory meal plans for a user with severe acne. The app provides 134 scientifically researched recipes (57 breakfast, 77 dinner), automatic shopping lists, and an educational section about acne treatment.

**Single-user app** — one person + partner, no multi-tenancy.

## Tech Stack

- **Backend:** Node.js + Express 5 (no build step, no TypeScript)
- **Frontend:** Vanilla HTML/CSS/JS single-page app (hash-based routing, no framework)
- **Data storage:** JSON files on disk (no database)
- **Auth:** bcryptjs password hashing, random token sessions stored in JSON
- **Deployment:** PM2 via `ecosystem.config.cjs`

## Project Structure

```
server.js                  — Express server, all API endpoints, weighted random algorithm
public/
  index.html               — SPA shell, login/setup forms, bottom nav
  style.css                — Mobile-first styles, all component styles
  app.js                   — Client-side routing, API calls, page renderers
data/
  recipes.json             — 134 recipes (generated by convert-recipes.js, DO NOT edit manually)
  tips.json                — 79 "tip of the day" entries with source citations
  learn.json               — Educational content structured in sections (supplements, skincare, hormones, lifestyle, gut)
  userdata.json            — Runtime user state (gitignored, created automatically)
convert-recipes.js         — Parses 6 markdown recipe files into data/recipes.json
ecosystem.config.cjs       — PM2 deployment config
*.md (recipe files)        — Source markdown files for recipes (not used at runtime)
```

## Key Architecture Decisions

### Recipe Data Pipeline
Source markdown files → `convert-recipes.js` → `data/recipes.json`. The conversion script handles 4 different markdown formats across 6 files. If recipe files are edited, re-run `node convert-recipes.js` to regenerate the JSON.

### Weighted Random Algorithm (server.js)
The weekly plan generator selects 7 breakfasts + 7 dinners sequentially (not independently) using:
1. **Time decay:** Recently cooked recipes get lower weights (0.02x within 7 days, scaling up to 1.0x after 56 days)
2. **Favorite boost:** Favorited recipes get 1.5x weight
3. **Exclusion:** Excluded recipes get weight 0 (never selected)
4. **Never-cooked bonus:** Uncooked recipes get 1.3x weight
5. **Ingredient overlap bonus (0.18x per shared ingredient):** After each selection, remaining candidates sharing ingredients with already-selected recipes get boosted. This reduces shopping list size by ~16%.

Selection is sequential: pick breakfast day 1, then dinner day 1 (with overlap from breakfast), then breakfast day 2 (with overlap from day 1), etc.

### Ingredient Normalization (server.js)
`normalizeIngredientName()` is used in two places:
1. Pre-computing ingredient sets for the overlap algorithm (at server startup)
2. Combining shopping list quantities

It strips: parenthetical info, comma suffixes, size adjectives (mittelgrosse/kleine/grosse), freshness adjectives (frischer/getrocknete), leading fractions, "zum Garnieren". Then applies a PLURAL_MAP for common German ingredient plurals.

### User Data (data/userdata.json)
```json
{
  "passwordHash": "bcrypt hash",
  "sessions": { "token": { "created": "ISO date" } },
  "currentPlan": { "weekStart": "YYYY-MM-DD", "days": [...] },
  "history": { "recipe-id": { "lastUsed": "ISO date", "useCount": N } },
  "favorites": ["recipe-id", ...],
  "excluded": ["recipe-id", ...],
  "shoppingChecked": ["normalized-ingredient-key", ...]
}
```

Loaded fresh on every request via `loadUserData()`, saved via `saveUserData()`. The `auth` middleware attaches `req.userData` and `req.saveUserData`.

### Auto-regeneration
When `GET /api/weekplan` is called and the current plan's `weekStart` doesn't match the current Monday, a new plan is automatically generated.

## API Endpoints

| Method | Path | Auth | Purpose |
|--------|------|------|---------|
| GET | /api/status | No | Check if password is set and if session is valid |
| POST | /api/setup | No | Set initial password (first-time only) |
| POST | /api/login | No | Authenticate, returns token |
| POST | /api/logout | Yes | Invalidate session |
| GET | /api/weekplan | Yes | Get current week plan (auto-generates if needed) |
| POST | /api/weekplan/generate | Yes | Force-generate new plan |
| POST | /api/weekplan/swap | Yes | Swap one recipe: `{ dayIndex, mealType }` |
| GET | /api/recipes | Yes | List all recipes (query: `type`, `search`) |
| GET | /api/recipes/:id | Yes | Single recipe detail |
| POST | /api/recipes/:id/favorite | Yes | Toggle favorite |
| POST | /api/recipes/:id/exclude | Yes | Toggle exclude |
| GET | /api/shopping-list | Yes | Shopping list from current plan |
| POST | /api/shopping-list/check | Yes | Check/uncheck item: `{ key, checked }` |
| GET | /api/tip | Yes | Tip of the day (rotates by day-of-year) |
| GET | /api/learn | Yes | Educational content |
| GET | /api/stats | Yes | Recipe counts and user stats |

Auth is via `x-auth-token` header. Frontend stores token in localStorage.

## Frontend Routing (public/app.js)

Hash-based SPA: `#plan`, `#recipes`, `#recipe/{id}`, `#shopping`, `#learn`. The `route()` function reads the hash and calls the appropriate `render*()` function which fetches data from the API and builds HTML via template literals.

## Recipe JSON Schema (data/recipes.json)

```json
{
  "id": "b-01",
  "type": "breakfast|dinner",
  "name": "Recipe Name",
  "category": "Overnight Oats",
  "cuisine": "Franzoesisch/Mediterran",
  "prepTime": 5,
  "ingredients": [
    { "text": "180 g Haferflocken (kernig)", "amount": 180, "unit": "g", "name": "Haferflocken" }
  ],
  "instructions": ["Step 1 text", "Step 2 text"],
  "skinBenefits": "Why this is good for skin..."
}
```

ID prefixes: `b-` (breakfast file 1), `b2-` (breakfast file 2), `d1-` through `d4-` (dinner files 1-4).

## Common Tasks

- **Add/edit recipes:** Edit the source `.md` files, then run `node convert-recipes.js`
- **Add tips:** Edit `data/tips.json` directly
- **Add learn content:** Edit `data/learn.json` directly
- **Start dev server:** `node server.js` (port 3000)
- **Deploy:** `pm2 start ./ecosystem.config.cjs --time`
- **Reset user data:** Delete `data/userdata.json`

## Important Notes

- Express 5 is used — wildcard catch-all route uses `/{*splat}` syntax, not `*`
- All text content uses ASCII-safe German (ae/oe/ue instead of umlauts) because the recipe source files were generated that way
- The `enrichRecipe()` / `enrichPlan()` helpers add `isFavorite` and `isExcluded` to recipe objects before sending to the frontend
- Shopping list ingredient categorization is keyword-based via `INGREDIENT_CATEGORIES` mapping in server.js
- The `css/` and `js/` directories in the project root are empty leftovers and not used
